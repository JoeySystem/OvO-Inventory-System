# 数据迁移操作步骤

## 1. 目的

本步骤用于将当前电脑上的物料管理系统数据，完整迁移到实际部署的电脑上。

本次迁移的目标是：

- 保留当前系统中的业务数据
- 让部署电脑启动后直接看到相同的数据
- 不重新手工录入物料、库存、单据、BOM、SOP、工单等内容

---

## 2. 本次会迁移哪些数据

当前系统的核心业务数据主要在 SQLite 数据库中，包括：

- 物料主数据
- 仓库资料
- 当前库存
- 入库 / 出库 / 调拨 / 盘点单据
- 库存流水
- BOM / SOP
- 生产工单 / 生产异常
- 历史采购记录
- 供应商价格本
- 一致性治理 / 供应治理 / 异常治理台账
- 用户、权限、操作日志

---

## 3. 当前电脑上的数据位置

当前项目默认数据目录：

- [data](/Users/Joey/CodexProjects/inventory-system-codex/data)

关键文件：

- [inventory.db](/Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db)
- [inventory.db-wal](/Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db-wal)
- [inventory.db-shm](/Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db-shm)
- [sessions.db](/Users/Joey/CodexProjects/inventory-system-codex/data/sessions.db)

说明：

- `inventory.db` 是主数据库
- `inventory.db-wal` 和 `inventory.db-shm` 是 WAL 模式下的辅助文件
- `sessions.db` 是登录会话库，不是核心业务数据，但建议一起带走

---

## 4. 迁移前准备

在开始前，先确认：

1. 当前系统没有人在继续录单
2. 当前电脑和部署电脑都可访问项目目录
3. 部署电脑已经具备可运行当前系统的环境
4. 部署电脑上已经有代码目录，但可以替换数据库

---

## 5. 推荐迁移方式

推荐使用：

- **停服务后拷贝数据库文件**

这是最稳的方式，能避免数据库正在写入时，出现主库和 WAL 文件不一致的问题。

---

## 6. 当前电脑上的操作步骤

## 6.1 停止当前系统

先停止当前电脑上的系统服务，确保数据库不再继续写入。

如果当前是终端直接运行：

```bash
Ctrl + C
```

如果是脚本或服务方式启动，请按实际启动方式停掉。

目标：

- 停机后不再有新的单据、库存、日志继续写入数据库

---

## 6.2 确认数据库文件存在

确认以下文件存在：

- [inventory.db](/Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db)
- [inventory.db-wal](/Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db-wal)
- [inventory.db-shm](/Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db-shm)

如果系统已经完全停稳，有时 `-wal` 和 `-shm` 可能不存在，这也正常。

---

## 6.3 打包要迁移的数据文件

建议把以下文件一起复制出来：

- `inventory.db`
- `inventory.db-wal`
- `inventory.db-shm`
- `sessions.db`

如果你想先做一份中转包，可在当前电脑上新建一个临时目录，例如：

```bash
mkdir -p /tmp/ovo-data-transfer
cp /Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db /tmp/ovo-data-transfer/
cp /Users/Joey/CodexProjects/inventory-system-codex/data/sessions.db /tmp/ovo-data-transfer/ 2>/dev/null || true
cp /Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db-wal /tmp/ovo-data-transfer/ 2>/dev/null || true
cp /Users/Joey/CodexProjects/inventory-system-codex/data/inventory.db-shm /tmp/ovo-data-transfer/ 2>/dev/null || true
```

然后再把这个目录拷到部署电脑。

---

## 7. 部署电脑上的操作步骤

## 7.1 先停止部署电脑上的系统

在部署电脑上，先停止当前正在运行的物料系统。

目的：

- 避免部署机上的旧数据库仍被占用
- 避免文件覆盖时出错

---

## 7.2 找到部署电脑的数据目录

根据部署电脑上的 `.env` 配置，找到它的数据库位置。

常见情况：

```env
DB_PATH=D:\\OvO-System\\data\\inventory.db
SESSION_DB_DIR=D:\\OvO-System\\data
```

如果部署电脑还没改过，可能仍是：

```env
DB_PATH=data/inventory.db
SESSION_DB_DIR=data
```

此时数据目录一般就在项目目录下的：

- `data\\`

---

## 7.3 备份部署电脑现有数据库

在覆盖前，先把部署电脑自己的旧数据库备份一份。

例如：

- `inventory_before_migration.db`

这样即使迁移过程中出问题，也能回滚。

---

## 7.4 用当前电脑的数据覆盖部署电脑的数据

把当前电脑带过去的文件复制到部署电脑对应的数据目录中，覆盖以下文件：

- `inventory.db`
- `inventory.db-wal`（如存在）
- `inventory.db-shm`（如存在）
- `sessions.db`（可选）

建议覆盖规则：

- 核心必须覆盖：`inventory.db`
- 推荐一起覆盖：`inventory.db-wal`、`inventory.db-shm`
- 会话可选覆盖：`sessions.db`

如果不覆盖 `sessions.db`：

- 用户只需要重新登录
- 不影响业务数据完整性

---

## 7.5 启动部署电脑上的系统

数据库文件放好之后，再启动部署电脑上的系统。

如果是命令行：

```bash
npm start
```

如果是 Windows 部署脚本，则按你们现有方式启动，例如：

- `start.bat`
- 或 Windows 服务

---

## 8. 迁移后的验收步骤

启动完成后，在部署电脑上按以下顺序检查。

## 8.1 健康检查

打开：

- `http://localhost:3000/api/health`

预期结果：

- `status: healthy`
- `database: connected`

---

## 8.2 登录检查

确认：

- 能正常登录
- 登录后能进入首页

如果没有迁移 `sessions.db`，重新登录即可。

---

## 8.3 核心数据检查

至少检查以下内容是否还在：

1. 物料数量是否正常
2. 仓库资料是否存在
3. 当前库存是否正常
4. 历史采购记录是否存在
5. BOM / SOP 是否存在
6. 生产工单 / 生产异常是否存在
7. 入库 / 出库 / 调拨 / 盘点单据是否能查到

---

## 8.4 随机抽查几条关键业务数据

建议抽查：

- 一张采购收货单
- 一条历史采购记录
- 一个物料详情
- 一个 BOM
- 一张生产工单

确认页面能正常打开、明细能查到。

---

## 9. 迁移时的注意事项

## 9.1 不要在系统运行中直接拷贝主数据库

否则容易出现：

- 主库和 WAL 文件不一致
- 数据不完整
- 部署机启动后有缺失

---

## 9.2 不要把部署机的空库反向覆盖本机数据

迁移方向必须是：

- 当前电脑的数据
- 覆盖到部署电脑

不要反过来操作。

---

## 9.3 优先相信数据库，不要靠页面手工对比全部数据

迁移的核心对象是数据库文件。  
只要数据库文件正确、系统能正常启动，绝大多数业务数据就会跟着过去。

---

## 9.4 如果部署电脑路径不同，不影响迁移

因为项目已经支持数据库路径配置化：

- 路径不同没关系
- 只要 `.env` 里的 `DB_PATH` 指向对的位置即可

---

## 10. 推荐最小迁移清单

如果只带最关键的数据，最少要迁：

- `inventory.db`

如果想更稳，建议迁：

- `inventory.db`
- `inventory.db-wal`
- `inventory.db-shm`

如果还想保留登录态，再加：

- `sessions.db`

---

## 11. 一句话执行口径

**先停当前系统，拷走 `inventory.db`（建议连同 `-wal / -shm` 一起），到部署电脑先备份旧库，再覆盖新库，最后启动并做健康检查与业务抽查。**
